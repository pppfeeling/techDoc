🧩 `tsconfig.json`의 `paths` vs `vite.config.ts`의 `alias` - 왜 둘 다 필요할까요?

Vite와 TypeScript를 함께 사용할 때, `@/components/MyComponent`처럼 경로 별칭(Path Alias)을 사용하면 코드가 훨씬 깔끔해지죠! 하지만 이 별칭을 사용하려면 `tsconfig.json` 파일과 `vite.config.ts` 파일 두 군데에 설정을 추가해야 할 때가 많아요. "어휴, 귀찮게 왜 두 번이나..." 😩 라고 생각하실 수 있지만, 여기에는 아주 중요한 이유가 있답니다!

📜 `tsconfig.json`의 paths 역할

이 설정은 주로 TypeScript 컴파일러(TSC)와 IDE(예: VSCode)를 위한 거예요.

  • 타입 체킹 & 지능형 코드 완성: IDE가 `@/components` 같은 경로를 이해하고, 해당 경로에 있는 모듈의 타입을 정확히 추론해서 타입 오류를 미리 잡아주거나 코드를 자동으로 완성해주는 데 사용돼요. 👩‍💻
  • 컴파일 시 경로 인식: TypeScript 컴파일러가 소스 코드를 분석할 때 이 `paths` 정보를 참조해서 모듈 간의 의존성을 파악해요.

하지만! `tsconfig.json`의 `paths` 설정만으로는 실제 실행 환경(브라우저 또는 Node.js)에서 모듈을 불러올 때 이 별칭 경로가 실제 파일 경로로 자동 변환되지는 않아요. TypeScript는 언어적인 측면에서 경로를 이해할 뿐, 번들러나 실행 환경이 이해하도록 경로를 바꿔주지는 않거든요.

예시: tsconfig.json

{ "compilerOptions": { "baseUrl": ".", // paths 옵션의 기준 경로 "paths": { "@/*": ["src/*"] // "@"로 시작하면 "src/" 폴더를 보라는 의미 } // ... 다른 옵션들 } }

🛠️ `vite.config.ts`의 resolve.alias 역할

이 설정은 Vite 개발 서버 및 프로덕션 빌드 과정에서 사용돼요.

  • 모듈 해석(Resolution): Vite는 개발 서버를 실행하거나 프로덕션용으로 빌드할 때, `import Button from '@/components/Button'`과 같은 코드를 만나면, 이 `@/components/Button`이라는 별칭 경로를 실제 파일 시스템의 경로(예: `/프로젝트경로/src/components/Button.tsx`)로 변환해줘야 해요. `resolve.alias`가 바로 이 역할을 담당합니다! 🗺️
  • 번들링 시 경로 처리: Vite는 내부적으로 Rollup(그리고 esbuild)을 사용하여 코드를 번들링하는데, 이 과정에서도 `alias` 설정이 올바른 모듈을 찾는 데 사용돼요.

만약 이 설정이 없다면? Vite는 `@/components`가 도대체 어디를 가리키는지 알 수 없어서 "모듈을 찾을 수 없습니다 (Cannot find module)" 오류를 뿜뿜! 💥 하게 될 거예요.

예시: vite.config.ts

import { defineConfig } from 'vite' import react from '@vitejs/plugin-react' import path from 'path' // Node.js 'path' 모듈 export default defineConfig({ plugins: [react()], resolve: { alias: [ // 배열 형태로 설정 가능 { find: '@', // 찾을 문자열 (별칭) replacement: path.resolve(__dirname, 'src') // 실제 경로 } // { find: '@components', replacement: path.resolve(__dirname, 'src/components') } // 더 구체적인 별칭도 가능 ] // 또는 객체 형태로도 가능: // alias: { // '@': path.resolve(__dirname, 'src') // } } })

🔗 그래서, 왜 둘 다 필요할까요?

각 설정은 서로 다른 도구의 세계를 담당하기 때문이에요!

🧐 TypeScript 컴파일러 & IDE (코드 작성, 타입 체크)
➡️ tsconfig.json (`paths`)
🚀 Vite (개발 서버, 빌드)
➡️ vite.config.ts (`resolve.alias`)

TypeScript는 우리가 코드를 작성하고 타입을 검사하는 개발 시점에 경로를 이해해야 하고, Vite는 실제로 코드를 실행하고 빌드하는 런타임 및 빌드 시점에 경로를 해석해야 해요. 두 시점에서 모두 경로 별칭을 올바르게 이해시키려면 각각의 설정 파일에 명시해줘야 하는 거죠. 마치 서로 다른 언어를 사용하는 두 사람에게 각각 통역사가 필요한 것과 같아요! 🌐

만약 하나만 설정하면 어떻게 될까요?

  • tsconfig.json에만 `paths` 설정: IDE에서는 경로를 잘 인식하고 타입 에러도 없지만, Vite 실행/빌드 시 "모듈 찾을 수 없음" 오류 발생! 😭
  • vite.config.ts에만 `alias` 설정: Vite는 잘 동작하지만, IDE에서 해당 경로를 인식 못해 타입 에러처럼 보이거나 자동완성 기능 사용 불가! 🙄

💡 꿀팁! 두 설정을 한 번에 관리하는 방법

매번 두 파일을 수동으로 동기화하는 것은 번거롭고 실수를 유발할 수 있어요. 이럴 때 vite-tsconfig-paths 같은 플러그인을 사용하면 편리해요! 이 플러그인은 `tsconfig.json` 파일의 `paths` 설정을 자동으로 읽어서 Vite의 `alias` 설정을 생성해준답니다. ✨

설치는 간단해요:

# npm npm install --save-dev vite-tsconfig-paths # yarn # yarn add --dev vite-tsconfig-paths # pnpm # pnpm add -D vite-tsconfig-paths

그리고 `vite.config.ts` 파일에 플러그인을 추가해주면 끝!

vite.config.ts (vite-tsconfig-paths 사용)

import { defineConfig } from 'vite' import react from '@vitejs/plugin-react' import tsconfigPaths from 'vite-tsconfig-paths' // 플러그인 import export default defineConfig({ plugins: [ react(), tsconfigPaths() // 플러그인 추가! ] // resolve.alias 설정은 이제 필요 없어요! (플러그인이 자동으로 해줌) })

이렇게 하면 `tsconfig.json`의 `paths`만 잘 관리해주면 Vite 설정은 신경 쓰지 않아도 돼서 훨씬 편해져요! 😊

이번 설명은 여기까지입니다! `tsconfig.json`의 `paths`와 `vite.config.ts`의 `alias`가 왜 각각 필요한지, 그리고 어떻게 함께 작동하는지에 대해 이해가 되셨나요?
